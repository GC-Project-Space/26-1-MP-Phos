# 지표 대시보드: Phos (모바일 인생네컷)

**날짜**: 2026-03-20  
**범위**: 제품 전체 (MVP v1: Live Booth + 메이킹 영상 + QR 사진/영상 다운로드 + 프라이버시 흐름)  
**단계**: 출시 전 -> 출시 초기 전환 구간  
**주요 목표**: `재미 중심 경험`을 2분 내 완주하고 결과물을 저장/공유하게 만드는 것

---

## 1) 무엇을 측정하는가 (1단계)

- **제품/기능 영역**: Live Booth 촬영, 렌더링, QR 다운로드, 프라이버시 제어(동의/삭제/내보내기)
- **현재 비즈니스 목표 / OKRs (도출값)**:
  - 첫 세션 완주율 >= 70%
  - 공유 전환율 >= 35%
  - 다운로드 성공률 >= 97%
  - 세션 실패율 < 3%
- **현재 상태**: 지표 정의는 있으나 계측 체계/대시보드/임계치 운영은 미정
- **제안 분석 스택**:
  - 제품 분석: PostHog 또는 Amplitude
  - 데이터 웨어하우스/BI: BigQuery + Metabase (또는 Looker)
  - 운영 알림: Grafana/Datadog + Slack/PagerDuty

## 2) 지표 프레임워크 (2단계)

## 노스스타

**노스스타 지표**: 2-Minute Keepsake Secured Rate (KSR)

**정의**:

```text
KSR = (2분 안에 capture -> render를 완료하고 최종 결과물을 최소 1개 확보한 세션 수)
      / (`sessions_started` 수)

"secure"는 `local_save_succeeded` 또는 `photo_download_succeeded` 또는 `video_download_succeeded`를 뜻함
집계 창: 일간, 추세 확인용 최근 7일 이동 뷰
```

**이 노스스타를 쓰는 이유**

- 사용자 가치(결과물을 내 손에 확보)와 제품 목표(재미+완주)를 직접 반영
- QR 같은 전달 채널에 과도하게 종속되지 않아 운영 안정성이 높음
- 퍼널 단계별 레버가 명확해 팀별 액션으로 연결 가능

**목표값 (출시 전 가설값 -> 보정값)**

- 소프트 런치 2주차: >= 30%
- 출시 D+30: >= 45%
- 출시 D+90: >= 55%

**보정 규칙 (출시 전 -> 출시 초기)**

- 하루 세션 모수가 50 미만이면 퍼널 지표는 정상/경고/위험 대신 `보정 중` 상태로 둔다.
- 최근 7일 삭제/내보내기 요청이 20 미만이면 프라이버시 SLA 지표는 `보정 중` 상태로 둔다.
- 퍼널 지표는 트래픽이 안정화될 때까지 rolling 7-day average를 사용한다.
- 시간 단위 알림은 크래시 비율, 렌더 지연시간, 프라이버시 침해 지표만 대상으로 한다.
- `Expired Asset Access Rate > 0` 또는 `trainingUsed=true`인데 유효하지 않은 동의가 있는 경우는 즉시 위험 상태다.

**에스컬레이션 정책**

- 경고: 제품/개발 온콜 채널에 알림, 24시간 내 원인 분석
- 위험: 즉시 PagerDuty, 2시간 내 완화 조치/롤백 여부 결정

### 구현 메모

- **데이터 소스**:
  - App event log: `session_started`, `camera_ready`, `capture_completed`, `render_succeeded`, `app_backgrounded`, `session_restored`
  - App event log (save): `local_save_tapped`, `local_save_succeeded`, `local_save_failed`
  - Share web log: `qr_opened`, `photo_download_started/succeeded/failed`, `video_download_started/succeeded/failed`, `asset_access_after_expiry_blocked`
  - Privacy log: `consent_updated`, `export_requested/completed`, `deletion_requested/completed`
- **필수 속성**:
  - `sessionId`, `deviceTier`, `networkType`, `elapsedMs`, `errorCode`
  - Privacy 메타: `retentionExpiresAt`, `trainingUsed`, `consentVersion`, `deletionStatus`
- **갱신 주기**:
  - 운영 상태: 5~15분
  - 제품 퍼널: 시간 단위
  - 리더십 대시보드: 일간 스냅샷
- **도구 권장안**:
- 제품 퍼널/코호트: PostHog/Amplitude
  - SQL 대시보드: BigQuery + Metabase
  - 알림: Datadog/Grafana + Slack + PagerDuty

### 리뷰 주기

- **일간**: 노스스타 지표 + 크래시/다운로드/프라이버시 상태 지표 확인
- **주간**: 입력 지표 드라이버 리뷰 + 실험 결과 공유
- **월간**: 노스스타 지표와 목표 갭 분석, 팀별 액션 재할당
- **분기별**: 지표 프레임워크 재검토(지표 폐기/추가)

## 5) 출시 보정 계획 (출시 전 필수)

출시 전에는 임계치가 가설값이므로 아래 순서로 보정합니다.

1. **Week 0**: 이벤트 스키마 고정 + QA 시뮬레이션 데이터로 수집 검증
2. **Week 1**: 소프트 런치 데이터로 기준선 산출 (P50/P95, 실패 코드 상위 5개)
3. **Week 2**: 정상/경고/위험 임계치 1차 조정
4. **Week 4**: NSM/입력 지표 상관관계 검증 후 노스스타 목표 재설정

## 6) 외부 벤치마크 메모 (검증된 기준 vs 가설값)

| 지표 영역                      | 베이스라인 유형          | 참조 범위                                  | 활용 방식                                                    |
| ------------------------------ | ------------------------ | ------------------------------------------ | ------------------------------------------------------------ |
| 모바일 리텐션 (D1/D7/D30)      | 검증된 업계 기준선       | Social/consumer 앱은 초기 리텐션 분산이 큼 | 절대 목표로 고정하지 않고 소프트 런치 기준선 비교에 사용     |
| 크래시 없는 세션 비율          | 검증된 안정성 기준선     | 상위 모바일 앱은 99%+ crash-free 지향      | 우리 지표는 `Session Crash Rate`로 역변환해 운영 알림에 사용 |
| DAU/MAU 고착도                 | 검증된 프레임워크 기준선 | 카테고리별 편차 큼 (social > utility)      | Phos는 이벤트형 사용이라 과도 목표 설정 금지                 |
| 공유/바이럴 계수               | 혼합형 (검증값 + 가설값) | 앱/루프 구조 따라 편차 큼                  | 초기 단계에서는 참고만 하고 핵심 운영지표는 아님             |
| 가치 체감까지 걸리는 시간 기준 | 가설 기반 팀 정의값      | 제품 플로우 특화 기준 필요                 | `2-minute` 경험 목표를 NSM과 퍼널에 반영                     |

주의사항:

- 출시 초기(표본 작음)에는 벤치마크 절대치보다 `추세`와 `이상징후 탐지`를 우선
- 공개 벤치마크 수치는 카테고리/국가/채널 편향이 있어 직접 목표치로 쓰지 않음
