## 1) 테스트 계획 개요 (1단계)

이 문서는 `Phos` MVP의 QA 실행 시나리오입니다.  
목표는 정상 흐름, 실패 흐름, 프라이버시 흐름을 모두 검증하는 것입니다.

테스트 우선순위:

- `P0`: 촬영, 렌더, 저장, 다운로드, 프라이버시 약속
- `P1`: 저사양 안정성, 중단 복구
- `P2`: 후속 우선순위 준비 항목

## 2) MVP 테스트 시나리오 (2단계)

### 테스트 시나리오 1. 세션 시작 및 카메라 준비

**테스트 목적:** Live Booth 시작 시 세션이 생성되고 카메라가 사용 가능한 상태가 되는지 검증한다.
**시작 조건:** 앱 설치됨 / 카메라 권한 허용 가능 / 네트워크 정상
**사용자 역할:** 사용자
**테스트 단계:** Live Booth 진입 -> 프레임 선택 -> 카메라 사용 가능 상태 확인 -> 실패 시 재시도 확인
**기대 결과:** 익명 세션 생성, 프레임 연결, `session_started`/`camera_ready` 기록, 실패 시 멈춘 상태 대신 재시도 제공

### 테스트 시나리오 2. 카운트다운 기반 다중 촬영 완료

**테스트 목적:** 카운트다운 기반 촬영이 선택한 프레임 타입에 맞춰 끝까지 완료되는지 검증한다.
**시작 조건:** 활성 세션 존재 / 프레임 선택 완료
**사용자 역할:** 사용자
**테스트 단계:** 촬영 시작 -> 각 컷 전 카운트다운 확인 -> 모든 컷 수행 -> 검토 단계 이동 확인
**기대 결과:** 각 컷 전 카운트다운 표시, 컷 수 일치, 각 컷 저장, `capture_completed` 기록

### 테스트 시나리오 3. 메이킹 영상 실패 시에도 사진 흐름 계속 진행

**테스트 목적:** 메이킹 영상 기록 실패가 포토 스트립 흐름을 막지 않는지 검증한다.
**시작 조건:** 활성 세션 존재 / 영상 기록 실패를 유도할 수 있는 환경
**사용자 역할:** 사용자
**테스트 단계:** Live Booth 시작 -> 메이킹 영상 실패 유도 -> 사진 컷 촬영 계속 가능 여부 확인 -> 안내 문구 확인
**기대 결과:** 사진 흐름 계속 진행, 자산 분리 규칙 유지, `making_video_failed` 기록, 명확한 안내 표시

### 테스트 시나리오 4. 컷 검토 및 스트립 슬롯 순서 변경

**테스트 목적:** 사용자가 촬영 후 컷을 보고 순서를 바꿀 수 있는지 검증한다.
**시작 조건:** 촬영 완료 세션 존재
**사용자 역할:** 사용자
**테스트 단계:** 검토 화면 진입 -> 전체 컷 확인 -> 스트립 슬롯 순서 변경 -> 렌더 단계 반영 여부 확인
**기대 결과:** 전체 컷 노출, 순서 변경 가능, 저장된 순서 반영, 편집 없이 렌더 진행 가능

### 테스트 시나리오 5. 간단한 사진 편집 적용

**테스트 목적:** MVP 범위의 빠른 편집만 적용되고, 실패 시 기존 선택 상태가 보존되는지 검증한다.
**시작 조건:** 검토 가능한 세션 존재
**사용자 역할:** 사용자
**테스트 단계:** 프리셋 필터 적용 -> 텍스트 오버레이 1개 적용 -> 프레임 크롭 적용 -> 편집 실패 상황 유도
**기대 결과:** MVP 범위 편집만 제공, 세션 상태 갱신, 편집은 선택 사항, 실패해도 컷/순서 보존

### 테스트 시나리오 6. 재시도 가능한 최종 포토 스트립 렌더

**테스트 목적:** 렌더 성공과 실패 재시도 흐름을 검증한다.
**시작 조건:** 선택된 컷과 프레임이 준비된 세션 존재
**사용자 역할:** 사용자
**테스트 단계:** 렌더 시작 -> 성공 시 최종 포토 스트립 자산 확인 -> 실패 환경에서 재시작 -> 재시도 경로 확인
**기대 결과:** 성공 시 최종 포토 스트립 자산 저장, `render_succeeded` 또는 실패 이벤트 기록, 재시도 제공, 결과가 저장/다운로드 흐름에 연결됨

### 테스트 시나리오 7. 앱 내 결과 화면에서 사진과 영상 저장

**테스트 목적:** 계정 없이 앱 내 결과 화면에서 존재하는 자산을 개별 저장할 수 있는지 검증한다.
**시작 조건:** 렌더 완료 세션 존재
**사용자 역할:** 사용자
**테스트 단계:** 결과 화면 진입 -> 영상 존재 시 영상 저장 시도 -> 사진 저장 시도 -> 만료/사용 불가 메시지 확인
**기대 결과:** `result_view_opened` 기록, 개별 저장 동작 제공, 계정 없이 저장 가능, 만료/사용 불가 안내 제공

### 테스트 시나리오 8. 최종 결과물 로컬 저장

**테스트 목적:** 로컬 저장 시작, 성공, 실패 흐름과 이벤트 기록을 검증한다.
**시작 조건:** 최종 결과물 화면 준비됨
**사용자 역할:** 사용자
**테스트 단계:** 로컬 저장 버튼 클릭 -> 성공 메시지 확인 -> 실패 환경 재시도 -> 세션 자산 무결성 확인
**기대 결과:** `local_save_tapped` 기록, 성공 시 `local_save_succeeded` 기록, 실패 시 자산 무손상

### 테스트 시나리오 9. 서비스 이용 동의와 선택적 데이터 활용 동의 분리

**테스트 목적:** 필수 서비스 이용 동의와 선택적 데이터 활용 동의가 분리 동작하는지 검증한다.
**시작 조건:** 새 세션 시작 가능
**사용자 역할:** 사용자
**테스트 단계:** 동의 화면 열기 -> 필수 동의만 수락 -> 선택 동의 비활성 유지 -> 다음 단계 가능 여부 확인
**기대 결과:** 필수/선택 분리 표시, `trainingOptIn` 기본 false, 필수 동의 없이는 진행 불가, 동의 스냅샷 저장

### 테스트 시나리오 10. 내보내기 및 삭제 요청

**테스트 목적:** 내보내기/삭제 요청이 보관 규칙과 생명주기 규칙에 맞게 동작하는지 검증한다.
**시작 조건:** 세션이 아직 `deleted`가 아님
**사용자 역할:** 사용자
**테스트 단계:** 내보내기 요청 생성 -> 내보내기 상태 확인 -> 삭제 요청 생성 -> 삭제 후 자산 접근 차단 상태 확인
**기대 결과:** 내보내기 요청은 `deleted` 이전에만 생성되고, 삭제 요청은 보관 기간 내에서만 생성되며, 완료 후 자산 접근이 차단되고 관련 이벤트가 기록된다

### 테스트 시나리오 11. 최소 세션 API 계약 검증

**테스트 목적:** MVP에 필요한 엔드포인트와 리소스 경로가 문서 계약대로 존재하는지 검증한다.
**시작 조건:** API 테스트 환경 준비됨
**사용자 역할:** QA / 백엔드 테스터
**테스트 단계:** 세션 생성/조회/수정 호출 -> 프레임 목록/단건 조회 호출 -> 세션 범위 리소스 경로 확인 -> 문서에 없는 커스텀 메서드 유무 확인
**기대 결과:** sessions/frames/세션 범위 리소스가 문서와 일치하고, 커스텀 메서드는 `:render`/`:finalize`만 존재하며, 사용자용 삭제는 `DeletionRequest`를 사용한다

### 테스트 시나리오 12. 보관 및 프라이버시 메타데이터 강제

**테스트 목적:** 48시간 보관, 메타데이터 포함, PII 미기록, 동의 불변조건을 검증한다.
**시작 조건:** 세션 생성 및 만료 시뮬레이션 가능
**사용자 역할:** QA / 프라이버시 테스터
**테스트 단계:** 새 세션 생성 및 `retentionExpiresAt` 확인 -> 만료 후 자산 접근 시도 -> 응답 메타데이터와 로그 확인
**기대 결과:** 모든 세션이 `createdAt + 48h` 보관 기간을 보유하고, 만료 후 `deleted` 상태가 되며, 프라이버시 메타데이터를 포함하고, 원본 자산 URL/직접 PII를 저장하지 않으며, `trainingUsed=true`는 유효한 동의 스냅샷이 있을 때만 허용된다

### 테스트 시나리오 13. 저사양 프리셋 자동 적용

**테스트 목적:** 저사양 기기에서 저사양 프리셋이 자동 적용되어 촬영/영상 흐름이 유지되는지 검증한다.
**시작 조건:** 저사양 기기 또는 동등한 시뮬레이션 환경
**사용자 역할:** 사용자 / QA
**테스트 단계:** 저사양 기기에서 Live Booth 시작 -> 프리셋 선택 결과 확인 -> 다중 컷 촬영 진행 -> 메이킹 영상 기록 동작 확인
**기대 결과:** 기본 프리셋 또는 저사양 프리셋 자동 선택, 해상도 또는 처리 부하 감소, 수동 기술 설정 없이 완료, 선택 프리셋 저장

## 3) 소프트 런치 안정화 시나리오 (3단계)

### 테스트 시나리오 14. 짧은 중단 후 미완료 세션 복구

**테스트 목적:** 짧은 중단 후 미완료 세션 복구가 안정적으로 동작하는지 검증한다.
**시작 조건:** `finalize` 전 활성 세션 존재
**사용자 역할:** 사용자
**테스트 단계:** 촬영/검토 도중 백그라운드 전환 -> 짧은 시간 내 복귀 -> 복구 성공/실패 경로 확인
**기대 결과:** 성공 시 미완료 세션 상태로 복귀하고, 실패 시 새 세션 안내가 표시되며, finalized 세션은 손상되지 않음

## 4) 후속 우선순위 시나리오 (4단계)

### 테스트 시나리오 15. 기존 사진으로 스트립 편집

**테스트 목적:** 후속 우선순위 Album Edit에서 기존 사진 가져오기와 기본 프레임 슬롯 편집이 가능한지 검증한다.
**시작 조건:** 후속 우선순위 빌드 또는 기능 플래그 활성화
**사용자 역할:** 사용자
**테스트 단계:** 기존 사진을 프레임 슬롯에 가져오기 -> 교체/크롭/회전 적용 -> 저장 가능한 결과 상태 확인
**기대 결과:** 기존 사진 가져오기 가능, 기본 프레임 슬롯 편집 동작, 이 시나리오는 MVP 출시 차단 항목이 아님

## 5) 데이터 직교성 비즈니스 로직 검증 (추가)

이 섹션은 `packages/db/prisma/schema/` 기반 직교성 이슈를 QA 실행 가능한 비즈니스 테스트로 문서화한다.

### 5.1 직교성 요구사항 표 (Source of Truth 기준)

| ID       | 문제 영역        | 비즈니스 규칙                                                           | Source of Truth                                           | 보조/스냅샷 데이터             | 불일치 발생 시 처리                                                     |
| -------- | ---------------- | ----------------------------------------------------------------------- | --------------------------------------------------------- | ------------------------------ | ----------------------------------------------------------------------- |
| ORTHO-01 | 선택 컷/순서     | 렌더 입력 컷과 순서는 관계형 선택 레코드 기준이어야 한다                | `SessionShotSelection(sessionId, assetId, position)`      | `Session.selectedShotAssetIds` | 관계형 레코드 기준으로 재생성하거나 요청을 거부한다                     |
| ORTHO-02 | 최종 산출물 식별 | 세션의 최종 사진/메이킹 영상은 세션 필드가 기준이다                     | `Session.finalPhotoAssetId`, `Session.makingVideoAssetId` | `Asset.assetRole`              | 세션 필드와 `assetRole`이 다르면 오류를 반환하고 정합성 이벤트를 남긴다 |
| ORTHO-03 | 동의 버전        | 감사 가능한 동의 버전은 동의 스냅샷 기준이어야 한다                     | `Consent.consentVersion`                                  | `Session.consentVersion`       | 최신 동의 스냅샷 기준으로 노출하고 세션 스냅샷은 동기화한다             |
| ORTHO-04 | 삭제 워크플로    | 삭제 요청 처리 상태와 세션 노출 상태는 동일 워크플로를 반영해야 한다    | `DeletionRequest.status`                                  | `Session.deletionStatus`       | 워커가 프로젝션을 재동기화하고 불일치 상태를 알림한다                   |
| ORTHO-05 | 보관 만료 정책   | 결과물 접근 가능 시간은 세션 보관 만료를 넘을 수 없다                   | `Session.retentionExpiresAt`                              | 없음                           | 만료 이후 접근 시도는 항상 거부한다                                     |
| ORTHO-06 | 편집 상태 경계   | 편집 JSON은 UI 편집값만 담아야 하며 도메인 정답 상태를 덮어쓰면 안 된다 | 정규 필드(`selectedShots`, `finalPhotoAssetId` 등)        | `Session.editState`            | 허용 키 외 편집 payload는 검증 실패 처리한다                            |

### 테스트 시나리오 16. 선택 컷 정합성 단일 기준 검증

**테스트 목적:** 컷 선택 데이터의 정답 기준이 `SessionShotSelection`으로 일관되게 동작하는지 검증한다.

**시작 조건:**

- `raw_shot` 에셋이 4개 이상 존재하는 활성 세션
- `PATCH /v1/sessions/{sessionId}`와 `POST /v1/sessions/{sessionId}:render` 호출 가능

**사용자 역할:** QA / 백엔드 테스터

**테스트 단계:**

1. `SessionShotSelection`에 `position` 0~3으로 컷을 저장한다.
2. `selectedShotAssetIds`를 의도적으로 다른 순서로 PATCH한다.
3. 렌더를 요청한다.
4. 렌더 입력으로 실제 사용된 컷 순서를 확인한다.

**기대 결과:**

- 렌더 입력 순서는 `SessionShotSelection.position` 기준으로 결정된다.
- `selectedShotAssetIds` 불일치가 감지되면 정합성 이벤트 또는 오류가 기록된다.
- 성공 응답에서는 단일 기준(관계형 선택)으로 정리된 결과만 노출된다.

### 테스트 시나리오 17. 최종 산출물 canonical 필드 검증

**테스트 목적:** 최종 사진/메이킹 영상의 canonical 참조가 세션 필드로 고정되는지 검증한다.

**시작 조건:**

- 렌더 성공 세션
- `final_photo`/`making_video` 역할 에셋이 존재

**사용자 역할:** QA / 백엔드 테스터

**테스트 단계:**

1. 렌더 직후 `Session.finalPhotoAssetId`, `Session.makingVideoAssetId`를 조회한다.
2. `Asset.assetRole`을 조회해 참조 대상과 비교한다.
3. 역할은 맞지만 세션 참조가 다른 데이터를 의도적으로 주입해 조회 API를 호출한다.

**기대 결과:**

- 사용자 노출 결과는 항상 세션의 canonical 필드를 따른다.
- canonical과 역할 분류가 다르면 `FAILED_PRECONDITION` 계열 오류를 반환한다.
- 운영 로그에 세션/에셋 불일치가 남는다.

### 테스트 시나리오 18. 동의 버전 스냅샷 동기화 검증

**테스트 목적:** 동의 버전의 정답이 동의 스냅샷에 있고 세션 스냅샷이 이를 추적하는지 검증한다.

**시작 조건:**

- 새 세션
- `POST /v1/sessions/{sessionId}/consents` 호출 가능

**사용자 역할:** QA / 프라이버시 테스터

**테스트 단계:**

1. `consentVersion=v1`로 동의를 생성한다.
2. `consentVersion=v2`로 재동의를 생성한다.
3. 세션 조회와 동의 목록 조회를 함께 수행한다.
4. 일부러 세션의 `consentVersion`만 오래된 값으로 두고 정책 검증 API를 호출한다.

**기대 결과:**

- 정책 판단은 최신 `Consent.consentVersion`을 기준으로 동작한다.
- `Session.consentVersion`은 조회 시 동기화되거나 불일치 경고가 노출된다.
- 동의 이력은 불변 레코드로 남는다.

### 테스트 시나리오 19. 삭제 요청 상태와 세션 상태 프로젝션 정합성 검증

**테스트 목적:** 삭제 워크플로 원본 상태와 세션 노출 상태가 레이스 상황에서도 일치하는지 검증한다.

**시작 조건:**

- 삭제 가능한 활성 세션
- 삭제 워커 실행 가능

**사용자 역할:** QA / 운영 테스터

**테스트 단계:**

1. 삭제 요청을 생성한다.
2. `DeletionRequest.status`와 `Session.deletionStatus`를 동시에 조회한다.
3. 워커 지연/재시도 상황을 만들어 상태 반영 순서를 흔든다.
4. 최종 완료 후 두 상태를 다시 비교한다.

**기대 결과:**

- 최종 일관 상태에서 두 값은 같은 워크플로 의미를 가진다.
- 일시 불일치가 발생하면 재동기화 작업이 수행되고 이벤트가 기록된다.
- `deleted` 확정 이후 자산 접근은 차단된다.

### 테스트 시나리오 20. 보관 만료 이후 결과물 접근 차단 검증

**테스트 목적:** 보관 만료 이후에는 어떤 결과물 접근도 허용되지 않는지 검증한다.

**시작 조건:**

- `retentionExpiresAt`가 설정된 세션
- 결과물 조회/저장 시도 API 사용 가능

**사용자 역할:** QA / 백엔드 테스터

**테스트 단계:**

1. 보관 만료 전 결과물 조회/저장을 시도한다.
2. 보관 만료 시간을 경과시킨다.
3. 보관 만료 이후 동일한 결과물 접근을 다시 시도한다.

**기대 결과:**

- 보관 만료 전 접근은 정책 범위 내에서 정상 동작한다.
- 보관 만료 이후 접근은 일관되게 거부된다.
- 거부 응답은 만료 원인을 명확히 포함한다.

### 테스트 시나리오 21. editState 도메인 경계 검증

**테스트 목적:** `editState`가 편집 전용 payload 범위를 지키는지 검증한다.

**시작 조건:**

- 편집 가능한 세션
- 세션 PATCH API 사용 가능

**사용자 역할:** QA / 백엔드 테스터

**테스트 단계:**

1. 허용 키(`filterPreset`, `textOverlay`, `cropToFrame`)로 PATCH한다.
2. 금지 키(`finalPhotoAssetId`, `selectedShotAssetIds`, `deletionStatus`)를 `editState`에 넣어 PATCH한다.
3. 렌더/조회 흐름에서 도메인 정답 필드 값이 변경되는지 확인한다.

**기대 결과:**

- 허용 키만 저장된다.
- 금지 키 포함 payload는 검증 실패로 거부된다.
- 편집 payload가 도메인 정답 필드를 덮어쓰지 않는다.

### 테스트 시나리오 22. 상태 전이 축 분리 검증

**테스트 목적:** `session.status`와 `session.deletionStatus`가 서로의 전이를 오염시키지 않는지 검증한다.

**시작 조건:**

- 렌더 완료 전/후 세션 2개
- 내보내기/삭제 요청 API 사용 가능

**사용자 역할:** QA / 백엔드 테스터

**테스트 단계:**

1. 렌더 전 세션에서 삭제 요청을 생성해 전이를 확인한다.
2. 렌더 완료 세션에서 내보내기 요청 후 삭제 요청을 수행한다.
3. 각 단계마다 `status`, `deletionStatus` 변화를 기록한다.

**기대 결과:**

- 렌더/완료 전이는 `session.status`에서만 관리된다.
- 내보내기/삭제 요청 전이는 `deletionStatus` 축에서만 반영된다.
- 한 축의 실패가 다른 축 상태를 임의로 되돌리지 않는다.

### 테스트 시나리오 23. 비동기 워커 경합 시 정합성 복구 검증

**테스트 목적:** 렌더/삭제 워커가 경합해도 최종 상태가 비즈니스 규칙으로 수렴하는지 검증한다.

**시작 조건:**

- 동일 세션에 대해 렌더 완료 직후 삭제 요청이 가능한 환경
- 비동기 작업 지연/재시도 시뮬레이션 가능

**사용자 역할:** QA / 운영 테스터

**테스트 단계:**

1. 렌더 완료 직후 삭제 요청을 생성한다.
2. 삭제/보관 워커 실행 순서를 교란한다.
3. 최종 세션/에셋 상태를 조회한다.

**기대 결과:**

- 최종 상태는 `session.status=deleted`, `deletionStatus=deleted`로 수렴한다.
- 결과물 접근은 모두 차단된 상태로 정리된다.
- 사용자 조회 API는 중간 불일치를 노출하지 않고 안정된 상태만 반환한다.
