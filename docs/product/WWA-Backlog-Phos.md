## 1) 백로그 개요 (1단계)

이 문서는 `Phos` MVP 구현을 위한 백로그 항목 모음입니다.  
각 항목은 Why-What-Acceptance 형식으로 정리했고, 한 스프린트 안에 논의하고 추정할 수 있는 크기로 맞췄습니다.

원칙:

- MVP에 직접 필요한 항목부터 먼저 적는다
- 소프트 런치 안정화 항목은 분리한다
- 후속 우선순위는 MVP에서 빼고 남긴다

## 2) MVP WWA 항목 (2단계)

### 항목 1. 세션을 시작하고 카메라를 준비한다

**이유:** 첫 진입이 느리면 재미 중심 제품의 핵심 가치가 깨집니다. 이 항목은 `빠른 완주`와 첫 세션 완주율 목표를 직접 지지합니다.

**무엇을:** 사용자가 Live Booth를 시작하면 익명 세션을 만들고, 프레임을 고른 뒤 바로 촬영 가능한 카메라 상태로 들어갑니다. 디자인은 추후 연결하되, 현재 논의의 핵심은 `세션 시작 -> 프레임 선택 -> 카메라 준비 완료`를 한 흐름으로 묶는 것입니다.

**인수 기준:**

- Live Booth 시작 시 익명 세션이 생성되고 `session_started`가 기록된다
- 사용자는 최소 1개의 4컷 프레임과 1개의 6컷 프레임 중 선택할 수 있다
- 선택한 프레임은 활성 세션에 저장된 뒤 카메라가 사용 가능한 상태가 된다
- 카메라가 사용 가능한 상태가 되면 앱은 `camera_ready`를 기록한다
- 카메라 준비 실패 시 사용자는 재시도 경로를 보고, 저장된 프레임은 세션에 유지된다

### 항목 2. 카운트다운 기반 다중 촬영을 진행한다

**이유:** 사용자는 포토부스처럼 리듬감 있게 촬영해야 결과를 쉽게 완성할 수 있습니다. 이 항목은 스트립 생성의 가장 기본 뼈대입니다.

**무엇을:** 선택한 프레임 타입에 맞춰 4컷 또는 6컷 촬영을 진행합니다. 각 컷 전에 카운트다운이 보이고, 촬영 결과는 현재 세션에 순서대로 저장됩니다.

**인수 기준:**

- 캡처 시작 후 각 컷 전 카운트다운이 표시된다
- 전체 컷 수는 선택한 프레임 타입과 일치한다
- 각 이미지 결과는 활성 세션의 일부로 저장된다
- 촬영 도중 중단되면 세션은 재시작 가능한 상태로 남는다

### 항목 3. 촬영 중 메이킹 영상을 기록한다

**이유:** `사진 + 메이킹 영상`은 Phos의 핵심 차별점입니다. 사진만 남기면 경쟁 제품과 차이가 줄어듭니다.

**무엇을:** Live Booth 촬영과 동시에 메이킹 영상을 기록합니다. 이 영상은 최종 포토 스트립과 다른 자산으로 세션에 연결됩니다.

**인수 기준:**

- 메이킹 영상은 Live Booth 촬영과 함께 시작된다
- 메이킹 영상은 최종 포토 스트립과 분리된 자산으로 저장된다
- 영상 기록 실패 시에도 사진 흐름은 계속된다
- 영상 실패 시 사용자에게 명확한 안내가 표시된다

### 항목 4. 촬영한 컷을 검토하고 순서를 정한다

**이유:** 사용자는 최종 포토 스트립에 어떤 사진이 어떤 순서로 들어가는지 제어할 수 있어야 만족도가 올라갑니다. 이는 재촬영보다 가벼운 품질 보정 수단입니다.

**무엇을:** 촬영 후 사용자는 모든 컷을 보고 스트립 슬롯 순서를 정합니다. 선택과 순서 정보는 이후 렌더링에 쓰일 세션 상태로 저장됩니다.

**인수 기준:**

- 촬영 후 사용자는 활성 세션의 모든 컷을 볼 수 있다
- 사용자는 컷을 스트립 슬롯에 배치하고 순서를 바꿀 수 있다
- 선택과 순서는 렌더에 쓰이는 세션 상태로 저장된다
- 사용자는 편집 없이 바로 렌더로 넘어갈 수 있다

### 항목 5. 간단한 사진 편집을 적용한다

**이유:** 너무 많은 편집은 흐름을 깨지만, 아무 편집도 없으면 결과물 만족도가 떨어질 수 있습니다. MVP는 빠른 완주를 해치지 않는 최소 편집만 제공해야 합니다.

**무엇을:** 사용자는 촬영 후 선택한 컷에 프리셋 필터, 텍스트 오버레이 1개, 프레임 크롭만 적용할 수 있습니다. 편집은 선택 사항이며 결과는 렌더용 세션 상태에 반영됩니다.

**인수 기준:**

- 사용자는 프리셋 필터, 텍스트 오버레이 1개, 프레임 크롭만 적용할 수 있다
- 각 편집 결과는 렌더에 쓰이는 세션 상태를 갱신한다
- 편집은 선택 사항이며 사용자는 바로 렌더로 넘어갈 수 있다
- 고급 편집 제어는 MVP 범위에 포함되지 않는다

### 항목 6. 최종 포토 스트립을 렌더링한다

**이유:** 렌더링이 안정적으로 끝나야 사용자가 결과물을 확보할 수 있습니다. 이 항목은 KSR과 다운로드 성공의 전 단계입니다.

**무엇을:** 현재 세션의 선택 컷과 프레임으로 최종 포토 스트립을 렌더링합니다. 실패 시에는 다시 시도할 수 있어야 합니다.

**인수 기준:**

- 활성 세션의 선택 컷과 프레임으로 렌더를 시작할 수 있다
- 렌더 성공 시 `render_succeeded`가 기록되고 최종 포토 스트립 자산이 세션에 저장된다
- 렌더 실패 시 실패 이벤트가 기록되고 재시도 경로가 제공된다
- 렌더 결과는 이후 로컬 저장과 QR 다운로드에 연결된다

### 항목 7. QR 페이지를 제공한다

**이유:** 결과물을 빠르게 받고 공유하는 것은 핵심 가치입니다. QR 페이지는 계정 없는 사용과 즉시 확보 흐름을 가능하게 합니다.

**무엇을:** 렌더 완료 후 세션은 QR 페이지를 노출하고, 존재하는 자산(사진, 영상)에 대해 각각 다운로드 액션을 제공합니다. 사용자는 계정 없이 이 페이지에 접근할 수 있습니다.

**인수 기준:**

- 렌더 완료 후 QR 페이지를 노출할 수 있다
- 페이지는 존재하는 자산(사진, 영상)에 대해 개별 다운로드 액션을 제공한다
- QR 페이지 접근 시 `qr_opened`가 기록된다
- QR 흐름은 계정 생성 없이 동작한다
- `shareLink.expiresAt`는 `retentionExpiresAt`를 넘지 않는다
- 자산이 만료되었거나 사용할 수 없으면 깨진 상태 대신 명확한 안내를 표시한다

### 항목 8. 최종 결과물을 로컬에 저장한다

**이유:** 사용자는 QR 페이지를 다시 열지 않아도 결과물을 로컬에 남기고 싶어 합니다. 로컬 저장은 결과물 확보 경험의 핵심입니다.

**무엇을:** 최종 결과물 화면에서 로컬 저장을 수행하고, 성공/실패를 알려줍니다. 저장 행동은 계측 가능해야 합니다.

**인수 기준:**

- 사용자는 최종 결과물 흐름에서 로컬 저장을 시작할 수 있다
- 저장 결과는 성공 또는 실패로 사용자에게 표시된다
- 시스템은 `local_save_tapped`와 `local_save_succeeded`를 기록할 수 있다
- 로컬 저장 실패가 세션 자산을 손상시키지 않는다

### 항목 9. 서비스 이용 동의와 선택적 데이터 활용 동의를 분리한다

**이유:** 프라이버시 신뢰는 제품 가치 자체입니다. 서비스 이용 동의와 선택적 데이터 활용 동의를 섞으면 신뢰와 이해도가 떨어집니다.

**무엇을:** 서비스 이용에 필요한 동의와 선택적 데이터 활용 동의를 분리합니다. `trainingOptIn=false`를 기본값으로 유지합니다.

**인수 기준:**

- 앱은 서비스 이용 동의와 선택적 데이터 활용 동의를 분리해 보여준다
- 새 세션에서 `trainingOptIn` 기본값은 `false`다
- 동의 상태는 `consentVersion`과 함께 세션에 저장된다
- 필수 서비스 이용 동의 없이는 사용자가 다음 단계로 진행할 수 없다

### 항목 10. 삭제 및 내보내기 요청을 지원한다

**이유:** 즉시 삭제/내보내기는 Phos의 프라이버시 약속을 실제 행동으로 보여주는 기능입니다. 단순 안내가 아니라 동작 가능한 제어가 필요합니다.

**무엇을:** 사용자는 보관 기간 안의 세션에 대해 내보내기 또는 삭제 요청을 보낼 수 있습니다. 삭제가 확정되면 공유 링크는 무효화됩니다.

**인수 기준:**

- 삭제되지 않은 세션에 대해 내보내기 요청을 만들 수 있다
- 보관 기간 안의 세션에 대해 삭제 요청을 만들 수 있다
- `exportRequest`는 삭제 확정 전에만 생성할 수 있다
- 삭제가 확정되면 해당 세션의 공유 링크는 무효화된다

### 항목 11a. 핵심 세션 및 렌더 계약을 출시한다

**이유:** 촬영과 렌더 흐름이 하나의 작은 안정 계약 위에서 움직여야 작은 팀이 재작업 없이 MVP를 끝낼 수 있습니다.

**무엇을:** PRD 범위 안에서 핵심 MVP 흐름이 실제로 쓰는 세션/프레임 조회, 세션 갱신, `:render`, `:finalize`만 먼저 고정합니다.

**인수 기준:**

- API는 `POST /v1/sessions`, `GET /v1/sessions/{sessionId}`, `PATCH /v1/sessions/{sessionId}`를 제공한다
- API는 `GET /v1/frames`와 `GET /v1/frames/{frameId}`를 제공한다
- 커스텀 메서드는 `POST /v1/sessions/{sessionId}:render`, `POST /v1/sessions/{sessionId}:finalize`만 제공한다

### 항목 11b. 세션 범위 공유 및 프라이버시 리소스를 출시한다

**이유:** 다운로드/프라이버시 요청은 필요한 범위만 별도 추적해야 작은 팀이 핵심 촬영 흐름과 병렬로 끝낼 수 있습니다.

**무엇을:** PRD에 정의된 세션 범위의 `assets`, `consents`, `shareLinks`, `exportRequests`, `deletionRequests` 경로만 제공합니다.

**인수 기준:**

- API는 `assets`, `consents`, `shareLinks`, `exportRequests`, `deletionRequests`의 세션 범위 경로를 제공한다
- 사용자용 삭제는 직접 세션 삭제가 아니라 `DeletionRequest`로 처리한다

### 항목 12. 보관 정책과 프라이버시 메타데이터를 강제한다

**이유:** 48시간 후 자동 삭제는 규칙이 아니라 제품 약속입니다. 이 약속이 시스템에서 강제되지 않으면 프라이버시 포지셔닝이 무너집니다.

**무엇을:** 각 세션에 48시간 보관 기간을 부여하고, 만료 시 삭제 상태로 전환합니다. 응답과 로그에는 필요한 프라이버시 메타데이터만 포함합니다.

**인수 기준:**

- 새 세션마다 `retentionExpiresAt = createdAt + 48h`가 설정된다
- `shareLink.expiresAt`는 항상 `retentionExpiresAt` 이하이다
- 만료 시간이 지나면 세션은 `deleted`로 전환되고 자산/공유 링크는 더 이상 사용할 수 없다
- 세션 관련 응답은 관련된 경우 `retentionExpiresAt`, `trainingUsed`, `consentVersion`, `deletionStatus`를 포함한다
- 로그는 `sessionId`, 시각, 액션, `consentVersion`, 상태 변화만 저장하고 원본 자산 URL이나 직접 PII는 저장하지 않는다
- `trainingUsed=true`는 해당 세션의 유효한 `consentVersion` 스냅샷이 있을 때만 허용된다

### 항목 13. 저사양 미디어 프리셋을 자동 적용한다

**이유:** 저사양 기기에서 실패하면 완주율과 신뢰가 함께 무너집니다. 사용자가 기술 옵션을 고르게 하지 않고도 안정성을 확보해야 합니다.

**무엇을:** 기기 성능을 기준으로 기본 프리셋 또는 저사양 프리셋을 자동 선택합니다. 이 프리셋은 다중 컷 촬영과 메이킹 영상 모두에 적용됩니다.

**인수 기준:**

- 캡처 시작 전 앱은 기본 프리셋 또는 저사양 프리셋을 자동 선택한다
- 저사양 프리셋은 비디오 해상도 또는 프레임 처리 부하 중 하나 이상을 낮춘다
- 사용자는 수동 기술 설정을 고를 필요가 없다
- 선택된 프리셋은 세션과 함께 분석용으로 저장된다

## 3) 소프트 런치 안정화 항목 (3단계)

### 항목 14. 짧은 중단 후 미완료 세션을 복구한다

**이유:** 소프트 런치에서는 작은 중단도 사용자 불만으로 이어집니다. 다만 이 항목은 V1 출시 게이트가 아니라 안정화 항목입니다.

**무엇을:** `finalize` 이전의 짧은 백그라운드 전환 후 재진입 시, 미완료 세션 복구를 시도합니다. 실패하면 새 세션 시작을 안내합니다.

**인수 기준:**

- 앱이 짧게 백그라운드로 전환된 뒤 재진입하면 미완료 세션 복구를 시도할 수 있다
- 복구에 성공하면 사용자는 미완료 세션 상태로 돌아간다
- 복구 실패 시 사용자는 새 세션 시작 안내를 본다
- 이 항목은 소프트 런치 안정화 항목으로 추적되고 V1 출시 게이트는 아니다

## 4) 후속 우선순위 항목 (4단계)

### 항목 15. 기존 사진으로 스트립을 편집한다

**이유:** 기존 사진 재활용은 장기 가치가 있지만, 첫 출시에서 핵심 완주 흐름보다 우선순위가 낮습니다. MVP를 가볍게 유지하기 위해 후속 우선순위로 둡니다.

**무엇을:** 사용자가 기존 사진을 프레임 슬롯에 넣고 기본 프레임 슬롯 편집을 적용하는 Album Edit 흐름입니다. MVP 이후에 계획합니다.

**인수 기준:**

- 사용자는 기존 사진을 프레임 슬롯에 가져올 수 있다
- 사용자는 교체, 크롭, 회전 수준의 기본 프레임 슬롯 편집을 할 수 있다
- 이 항목은 MVP v1 출시에 필수는 아니다
- 핵심 Live Booth, 프라이버시, 다운로드 흐름 안정화 이후 계획한다
