## 1) 문서 목표

이 문서는 현재 PRD, 사용자 스토리, 백로그에 흩어져 있는 요구사항을 실제 모바일 화면 설계 관점으로 재정리한 UI/UX 정의서입니다.

목표는 세 가지입니다.

- 제품 요구사항을 화면 흐름 기준으로 해석한다
- 디자인/개발/QA가 같은 UX 기준으로 일할 수 있게 한다
- 현재 프론트엔드 스캐폴드와 앞으로 구현할 MVP 화면 사이의 기준선을 만든다

## 2) 제품 UX 테제

### 경험 약속

Phos는 `2분 안에 촬영을 끝내고 결과물을 확보하는 재미 중심 모바일 포토부스`여야 합니다.

### 핵심 UX 가치

- 빠르게 시작한다
- 촬영 흐름이 끊기지 않는다
- 사진과 메이킹 영상을 함께 남긴다
- 계정 없이 결과물을 확보한다
- 프라이버시 제어가 짧고 명확하다

### UX 원칙

1. **첫 행동까지 빠르게**: 첫 진입 후 사용자는 최대한 적은 판단으로 촬영을 시작해야 한다.
2. **한 손으로도 이해 가능하게**: 핵심 액션은 한 손으로도 이해 가능해야 하며, 화면마다 주 행동은 하나여야 한다.
3. **도구보다 흐름 우선**: 고급 편집보다 리듬감 있는 촬영 완주를 우선한다.
4. **벌주지 말고 복구시키기**: 실패 시 처음부터 다시 배우게 하지 말고 재시도와 복구 경로를 준다.
5. **프라이버시는 제품의 일부**: 동의, 보관, 삭제, 내보내기는 법무 문구가 아니라 사용자 경험이어야 한다.

## 3) 주요 사용자와 맥락

### 주요 사용자

- 친구, 연인, 소규모 모임에서 짧은 시간 안에 결과물을 남기고 싶은 사용자
- 별도 가입이나 복잡한 편집 없이 바로 찍고 저장하고 싶은 사용자

### 보조 사용자

- 행사/이벤트에서 여러 사람이 결과물을 빠르게 확보하게 하고 싶은 사용자
- 프라이버시 민감도가 높아 데이터 보관과 삭제 통제를 중요하게 보는 사용자

### 사용 맥락

- 서서 빠르게 사용한다
- 촬영 현장은 밝기, 소음, 네트워크 상태가 일정하지 않다
- 사용자는 앱에 오래 머물 의도가 없고, 즉시 완주를 기대한다

## 4) MVP 경험 흐름

```text
진입
  -> Live Booth 시작 버튼
  -> 프레임 선택
  -> 카메라 준비 완료
  -> 카운트다운 연속 촬영
  -> 메이킹 영상 동시 기록
  -> 촬영 결과 검토 / 순서 조정
  -> 간단 편집(선택)
  -> 최종 렌더링
  -> 결과 확인
  -> 로컬 저장
  -> 필요 시 삭제 / 내보내기 요청
```

### 흐름 수준 요구사항

- 메인 플로우는 계정 생성 없이 완료 가능해야 한다
- `Live Booth 시작`은 즉시 촬영 개시가 아니라 익명 세션 생성 후 프레임 선택으로 진입시키는 시작 액션으로 정의한다
- 사용자는 어떤 단계에 있는지 항상 알 수 있어야 한다
- 각 단계의 주 CTA는 한 개만 강조한다
- 실패 상태는 `왜 실패했는지`, `지금 무엇을 할 수 있는지`를 바로 알려야 한다
- 결과물 확보 전까지 사용자가 시스템 상태를 추측하게 두지 않는다

## 5) 화면 요구사항

### 5.1 진입 / 홈

- **목표**: 제품 가치와 준비 상태를 빠르게 이해하고 `Live Booth 시작`으로 진입하게 한다.
- **필수 콘텐츠**: 제품명 `Phos`, 핵심 가치 한 줄, 이용 가능 프레임 요약, 준비 상태 요약, 주 CTA
- **UX 요구사항**: 브랜드와 핵심 액션이 첫 화면에서 동시에 보여야 하고, 소개보다 시작 가능 여부 판단에 집중해야 한다.
- **상태**: 기본 / 카메라 준비 불가 / 오프라인 또는 백엔드 준비 불가

### 5.2 프레임 선택

- **목표**: 4컷 또는 6컷 프레임을 2초 안에 이해하고 선택하게 한다.
- **필수 콘텐츠**: 프레임 썸네일, 프레임 이름, 컷 수, 활성 여부, 선택 완료 CTA
- **UX 요구사항**: 최소 1개의 4컷과 1개의 6컷 프레임 제공, 차이를 즉시 이해 가능, 선택 결과는 활성 세션에 저장 후 다음 단계 이동
- **상태**: 선택 전 / 선택됨 / 비활성 / 로딩/재시도

### 5.3 카메라 준비 / 촬영 시작

- **목표**: 카메라가 촬영 가능한 상태임을 확신하고 촬영을 시작하게 한다.
- **필수 콘텐츠**: 선택 프레임 요약, 현재 세션 상태, 카메라 준비 여부, 주 CTA: `촬영 시작`
- **UX 요구사항**: 사용 가능 상태 전 CTA 비활성, 준비 실패 시 재시도 경로 명확화, 프레임 선택 정보 유지

### 5.4 카운트다운 다중 촬영

- **목표**: 포토부스처럼 리듬감 있게 4컷 또는 6컷 촬영을 완료하게 한다.
- **필수 콘텐츠**: 현재 컷 번호, 다음 촬영까지 카운트다운, 진행률, 중단/재시작 가능 여부, 메이킹 영상 기록 상태
- **UX 요구사항**: 현재 위치 명확화, 카운트다운은 읽히되 흐름을 늦추지 않아야 함, 메이킹 영상 실패가 사진 흐름을 막지 않아야 함
- **상태**: 대기 / 카운트다운 / 촬영 중 / 일시 중단 / 영상 기록 실패 / 캡처 실패

### 5.5 컷 검토 및 순서 조정

- **목표**: 촬영한 컷 전체를 빠르게 훑고 최종 포토 스트립 구성을 결정하게 한다.
- **필수 콘텐츠**: 전체 컷 썸네일, 현재 스트립 슬롯 순서, 선택 상태, 렌더 CTA, 선택 편집 CTA
- **UX 요구사항**: 전체 스캔 가능, 순서 변경은 직접적이고 되돌릴 수 있어야 함, 편집 없이도 바로 렌더 가능

### 5.6 간단 편집

- **목표**: 흐름을 깨지 않는 최소 편집만 적용하게 한다.
- **허용 범위**: 프리셋 필터 / 텍스트 오버레이 1개 / 프레임 크롭
- **UX 요구사항**: 도구는 적고 빠르게 이해 가능, skip 명확, 편집 실패가 선택/순서를 손상시키면 안 됨, 고급 편집 UI는 MVP 제외

### 5.7 렌더

- **목표**: `최종 결과를 만드는 중`임을 이해하고 다음 행동을 바로 알게 한다.
- **필수 콘텐츠**: 렌더 진행 상태, 성공 시 결과 미리보기, 실패 시 재시도 CTA
- **UX 요구사항**: 시스템 상태를 숨기지 않고, 성공 후 결과가 저장 액션에 연결되어야 하며, 실패 시 재시도 가능해야 한다

### 5.8 결과 / 저장

- **목표**: 최종 포토 스트립과 메이킹 영상을 가장 확실한 방식으로 확보하게 한다.
- **필수 콘텐츠**: 최종 포토 스트립 미리보기, 메이킹 영상 존재 여부, `로컬 저장`, 자산별 저장 구분
- **UX 요구사항**: 로컬 저장 진입 경로를 단순화하고, 만료/부재 자산은 설명형 상태로 처리한다

### 5.9 프라이버시 동의

- **목표**: 서비스 이용 동의와 선택적 데이터 활용 동의를 헷갈리지 않고 이해하게 한다.
- **필수 콘텐츠**: 서비스 이용 동의, 선택적 데이터 활용 동의, 보관 기간, 삭제/내보내기 가능 안내
- **UX 요구사항**: 필수와 선택 항목을 시각적으로 분리하고, `trainingOptIn=false`가 기본 상태여야 하며, 긴 법률 문구보다 한 줄 요약을 먼저 보여준다

### 5.10 삭제 / 내보내기 제어

- **목표**: 자신의 세션 데이터를 직접 제어할 수 있다고 느끼게 한다.
- **필수 콘텐츠**: 현재 삭제 상태, 내보내기/삭제 요청 액션, 요청 이후 상태 안내
- **UX 요구사항**: 삭제 비가역성 명확화, 내보내기는 삭제 확정 전까지만 허용, 삭제 이후 자산 접근 차단 결과 설명

## 6) 정보 구조

### MVP 화면 맵

1. 진입 / 홈
2. 프레임 선택
3. 카메라 준비
4. 카운트다운 촬영
5. 컷 검토
6. 간단 편집
7. 렌더 진행
8. 최종 결과
9. 동의 / 프라이버시 제어
10. 삭제 / 내보내기 요청 화면

결과 화면과 삭제/내보내기 영역은 구현 단계에서 별도 화면 또는 패널로 분리될 수 있으나, UX 책임은 문서상 분리해서 관리한다.

### 내비게이션 규칙

- 메인 플로우는 선형 구조를 유지한다
- 이전 단계로 돌아갈 수 있지만 이미 생성된 세션 상태는 유지해야 한다
- 자동 복구 가능하면 복구 제안을 먼저 보여준다

## 7) 콘텐츠 요구사항

### 보이스와 톤

- 짧고 직접적이어야 한다
- 마케팅 문장보다 안내 문장 중심이어야 한다
- 기술 용어보다 행동 중심 언어를 사용한다

### 카피 규칙

- 화면당 제목은 하나
- 버튼 문구는 행동 동사 중심
- 에러 문구는 원인 + 다음 행동을 같이 제시
- 프라이버시 문구는 숨기지 말고 짧게 요약

예시:

- 좋음: `지금 3번째 컷을 찍어요`
- 좋음: `메이킹 영상 기록에 실패했어요. 사진 촬영은 계속할 수 있어요`
- 피해야 함: `예기치 못한 오류가 발생했습니다`

## 8) 시각 및 상호작용 요구사항

### 시각 방향

- 앱은 `재미 중심` 제품이지만 조작은 차분하고 명확해야 한다
- 결과물 중심 제품이므로 촬영 상태와 결과물 미리보기가 시각적 중심이어야 한다
- 프라이버시 관련 정보는 보조 정보가 아니라 신뢰 레이어로 보여야 한다

### 상호작용 요구사항

- 주요 액션은 엄지 닿는 영역 안에 배치한다
- 카운트다운, 진행률, 성공/실패 상태는 즉시 피드백해야 한다
- 모션은 분위기보다 방향성과 상태 전달에 써야 한다

## 9) 접근성 및 성능 요구사항

### 접근성

- 텍스트와 배경 대비는 모바일 야외 사용을 고려해 충분해야 한다
- 상태 변화는 색상만으로 구분하지 않는다
- 터치 타깃은 충분히 커야 한다
- 중요한 상태와 CTA는 스크롤 없이 인지 가능해야 한다

### 성능

- 카메라 진입은 체감 2초 내를 목표로 한다
- 저사양 기기에서는 자동 프리셋 조정이 적용되어야 한다
- 렌더/저장/다운로드는 장시간 무응답처럼 보이지 않아야 한다

## 10) MVP 비목표

- 고급 사진 편집 도구
- 복잡한 템플릿 커스터마이징
- 계정 기반 개인화 흐름
- Album Edit 본 기능
- 출시 게이트로서의 고도화된 세션 복구

## 11) 현재 레포 상태 기준 구현 메모

- 현재 모바일 앱은 `BoothHomeScreen`, `ExperienceOverview`, `SessionReadiness`, `InfoCard` 중심의 스캐폴드 상태다
- 현재 화면은 요구사항 설명용 홈에 가깝고, 실제 촬영 플로우 화면은 아직 구현 전이다
- 따라서 이 문서는 `현재 구현 설명서`라기보다 `현재 요구사항을 구현하기 위한 UX 기준 문서`로 사용해야 한다

## 12) 디자인/구현 인수 체크리스트

- 첫 화면에서 제품명, 핵심 가치, 시작 CTA가 동시에 보이는가
- 프레임 선택이 2초 안에 이해되는가
- 촬영 중 현재 컷과 진행 상태가 명확한가
- 메이킹 영상 실패가 사진 흐름을 막지 않는가
- 편집 없이도 렌더로 갈 수 있는가
- 결과 화면에서 저장 경로가 단순하고 명확한가
- 필수 서비스 이용 동의와 선택적 데이터 활용 동의가 시각적으로 분리되어 있는가
- 삭제/내보내기 제어가 실제 행동으로 이어지는가
- 만료/실패/복구 상태가 모두 설명 가능한가
